『Kubernetesで実践するクラウドネイティブDevOps』 読んだ

2023-11-09
ITインフラ, ネットワーク, 書物

読んだ理由 : DevOpsの思想と実践方法を身につけたい。あといい加減Kubernetesと友達になりたい。

オライリーのページ : https://www.oreilly.co.jp/books/9784873119014/

1章 クラウドでの革命

クラウドコンピューティング、DevOps、Docker、Kubernetesの歴史的経緯を記述している。

「システム」はもはや、自分のソフトウェアだけでは成り立ちません。自社開発のソフトウェアに加え、クラウドサービス、ネットワークリソース、ロードバランサ、監視、コンテンツ配信ネットワーク、ファイアウォール、DNS など幅広い要素で構成される上、これらすべてが緊密に相互接続され、相互に依存し合っています。ソフトウェアを書く者はシステム全体との関連性を理解しなければならず、システムを運用する者はソフトウェアが機能する仕組みや機能しない理由を理解しなければなりません

こういう文章はプログラマなりたてのころから見ているが、Webアプリ開発業務に携わりだして身をもって実感するようになった。

DevOps を導入するためには、ビジネスの文化を根本的に変革する必要があり、まずは経営幹部による戦略的なレベルから取り組みを始め、組織のあらゆる部分へと徐々に広めていくことが求められます。

まじ？弊社ではどうなってるんだっけ……。DevOpsの仕組み自体は存在するけど、文化面については考えたこともなかった。
てかサービス基盤チームがすごすぎて（アプリもゴリゴリに書ける）、インフラに関してはその人達に頼りっきりになっちまってる。頼り切りは言い過ぎかもしれんが、少なくともインフラ領域のスキルにおいて雲泥の差ではある。

Kubernetes の将来は、主としてマネージドサービスの領域にこそあると考えられます。例えば仮想化は、かつては熱狂的に迎えられた最新テクノロジでしたが、現在では単にユーティリティとなっています。ほとんどのユーザが、vSphere や Hyper-V などの仮想化プラットフォームを独自に実行するのではなく、クラウドプロバイダから仮想マシンをレンタルしています。

Azure Container Appsなどのように、その片鱗は現れてきている。

CNCF 傘下の最も知名度が高いプロジェクトは Kubernetes そのものですが、CNCF では Prometheus、Envoy、Helm、Fluentd、gRPC をはじめとして、クラウドネイティブエコシステムにおける他の重要なコンポーネントも数多く生み出し、推進しています。

知らなんだ。このうちPrometheusとEnvoyはお世話になってるのに。gPRCは触ったことないけど、JSONと違って型の不整合が起きにくそうで気になる。

3章 Kubernetes環境の選択

Kubernetes環境の構築はセルフホスティングで行うのかクラウドのマネージドサービスで行うのか。まあ当然マネージドサービスを使うべき（弊社はAKS）。

Kubernetes が複数のサーバを接続して 1 つのクラスタとすることは言うまでもないと思いますが

いやぁそれすら存じませんでした（あれ、今のおれにはこの本は時期尚早か？）。

何もない状態からKubernetes を本番の設定で運用し始めるためには、エンジニアの給与で約 100 万ドルが必要になると推定（https://twitter.com/copyconstruct/status/1020880388464377856 ）しています（「それでも目的を達成できないかもしれません」）。技術部門の責任者が Kubernetes のセルフホスティングを検討する際には、この数字が 1 つの判断基準となるでしょう。

セットアップで考慮すべき項目が大量に紹介されており、いくら金積んでもひとりじゃ無理ゲーなレベル。

Netflix が提供している Chaos Monkey のような自動化された回復力テストツールは、時折ランダムにノードや Pod を強制終了したり、ネットワーク接続を切断してくれるため、こうした取り組みに役立ちます。

そんなのあるのかよ。こういう分散システムの対障害性をテストすることをカオスエンジニアリングというらしい。

Kubernetes クラスタのセットアップと設定を支援するツールは数限りなく存在しており、そのほとんどが多かれ少なかれ、マウスによるクリック操作で、何の苦労もなく、すぐに使用できるソリューションであると謳っています。しかし残念ながら、著者らの見解では、こうしたツールの大半が解決するのは簡単な問題だけであり、難しい問題は無視するのが実情です。

Kubernetesに限った話じゃねーけどな。

EKS は他のクラスタインフラに加え、マスタノードにも課金します。そのため、クラスタの規模が同じ場合、Google や Microsoft のマネージド Kubernetes サービスより割高になります。

へえ意外。

「実行するソフトウェアを減らす」という哲学には 3 本の柱があり、いずれも時間をうまく使って競争相手に打ち勝つために役立ちます。

  1. 標準のテクノロジを選ぶ
  2. 差別化につながらない面倒な作業はアウトソーシングする
  3. 長続きする競争優位性を生み出す ―― Rich Archbold

この哲学は知らなかったけど内容は完全同意。

Kubernetes は標準プラットフォームであるため、例えば Google Kubernetes Engine で実行するために構築するアプリケーションやサービスは、他のいかなる認定 Kubernetes プロバイダのシステム上でも機能します。そもそも Kubernetes を使用すること自体が、ベンダロックインの回避に向けた大きな 1 歩となるのです。

てことはKubernetesを習得しとけば、知らんクラウド使ってる企業への転職ハードルが低くなるな。

関係なかったわ。まあええか。

kubectl run コマンドは実際には Pod を直接作成したわけではなく、その代わりに Deployment を作成しました。これを受けて、Deployment が Pod を起動したのです。

help見る限りだとkubectl runは直接podを生成してるように思える。上のほうでdeleteしても復活しなかったし。よくわからんが、仕様が変わったのかもしれない。

❯ k run --help

ユーザが Deployment を更新する際には、新しい ReplicaSet が作成されて新しい Pod を管理し、更新が完了した時点で、古い ReplicaSet とその Pod は終了されます。

こういう仕組みでServerStarterみたいなことをやってくれるわけか。

スケジューラの仕事は、スケジューリングされていない Pod のキューを監視し、そこから次のPod を取り出して、実行場所となるノードを見つけることです。

そういえばノードのことまだ理解できてない。

Service は Web プロキシやロードバランサのようなものと考えることができ、バックエンドである Pod のグループにリクエストを転送します（図4-2）。とはいえ、これは Web ポートに限定されません。Service では、スペックの ports 部分で定義される詳細にしたがって、任意のポートから他の任意のポートへトラフィックを転送できます。

前から思ってたが、Service以外にもっとましな名称あっただろ。

宣言的な YAML マニフェストを使用してアプリケーションを Kubernetes クラスタにデプロイする方法については、これで必要なことをすべて説明しました。しかし、こうしたファイルには多くの繰り返しがあります。例えば、名前の demo、Label セレクタの app: demo、ポートの 8888 は何度も出現します。

5章 リソースの管理

十分なリソース量を確保できるノードが存在しない場合、条件に合致するノードを利用できるようになるまで、Pod は pending（待機）状態のままとなります。

おれみたいなアホがハマりそうなトラブルだ。

spec.containers.resources.requestsがリソースの下限、spec.containers.resources.limitsが上限と考えていいのかな。

ヘルスチェックに関する特定のエンドポイントがアプリケーションに用意されていない場合には、/または任意の有効な URL をアプリケーション用に使用できます。とはいえ、この目的のためには/healthz というエンドポイントを作成するのが一般的な慣行となっています（最後に z を付けるのは、例えば健康情報に関するページなど、health がよく使われる既存のパスと競合するのを避けるためです）。

ほーlivenessProbe便利。弊社のk8sマニフェスト見たらこれを設定しているプロダクト少ない。悲しい。

コンテナに Readiness probe が設定されている場合、Kubernetes はこのプローブの結果が成功となるまでは、リクエストを送信せずに待機します。これは準備のできていないコンテナからユーザにエラーが返される事態を回避するためです。この動作が特に重要となるのはゼロダウンタイムアップグレードの場合です

readinessProbeのほうは弊社では結構な箇所で使われている。なぜだ。

minAvailabeで生存させるpodの最小数を決めたり、maxUnavailableでk8sがノードから退去させられるpodの総数を決めたり。

Namespace は関連するリソースをグループ化して、それらを使った作業を簡素化します。ただしフォルダと異なり、Namespace をネストさせることはできません。

残念なことなのかそうでもないのかよくわからない。

Namespace を削除すると、その中にあるリソースもすべて削除されます。削除コマンドの実行対象とする Namespace を間違えるのは、絶対に避けたい事態です

本書でフォルダのメタファを持ち出してたけど、削除にもあてはまるのか。

例えば prodNamespace の demo Service と通信したい場合には、次のように指定できます。 demo.prod

namespaceがトップドメイン、serviceがApexドメインみたいな扱いか。

Namespace 内の Pod について CPU とメモリの総使用量を制限することもできますが、著者らは推奨しません。こうした総量の設定が低すぎると、ワークロードが制限値に近づいた場合、想定外で特定困難な問題に直面する可能性が高くなります。逆に設定が高すぎると、そもそも制限する意味が薄れてしまいます。

逆にnamespace内のpod数には制限をかけるべきとしている。

LimitRange は、コンテナの所有者がリソース要求またはリソース制限を指定し忘れた場合に発生しうる問題を防止するための、最後の砦としてのみ使用するようにします。

オブジェクト指向プログラミングでよくあるハマりパターン。

Job オブジェクトは Kubernetes データベースに存在したままとなるため、完了した Job が相当な数に達すると、API の処理能力に悪影響を及ぼす可能性があります。

え、知らなかった。今のk8sバージョンだとどうなってるんだろう。
