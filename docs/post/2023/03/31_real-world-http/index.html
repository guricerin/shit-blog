『Real World HTTP 第2版』

1章 HTTP/1.0の世界 : 基本となる4つの要素

HTTPの成り立ちを解説。

HTTP

  • Hyper Text Transfer Protocol
  • 元々はWebブラウザとWebサーバー間の通信プロトコル。
    • リクエストを受け付け、適切なドキュメントを返すだけ。
  • 現在はその用途に限定されず、インターネットの基盤。
  • メソッドとパス
  • ヘッダー
  • ボディ
  • ステータス

RFC

  • 様々なプロトコルの定義書。
  • Request For Commentの由来
    • ネットワークはもともとアメリカの国防予算で作られたため、仕様を外部に公開できなかった。
    • 「品質向上のための意見を世界中から集める」という名目で仕様を公開することにした名残。

ヘッダー

  • format
    • field: value
  • 電子メールのプロトコルからパクられた。
  • filed名は大文字小文字の区別をしない。
  • X-からはじまるヘッダーは各アプリが自由に使用してよい。
    • 最近はRFCをなるべく発行して登録済みheaderを周知し、X-なしのヘッダーを普及させるような動きがある。
  • 正規化の方法はフレームワークによって異なる。
    • 同名ヘッダーのvalueを,区切りでひとつに連結した文字列にしたり、文字列の配列にしたり、単語の先頭を大文字にしたり。

MIMEタイプ

  • ファイル形式の識別子。
    • {大項目}/{詳細}
  • HTTPではContent-Typeヘッダーに格納される。
Content Sniffing
  • （ブラウザが）MIMEタイプではなく中身を見てファイルの種類を推測すること。
  • 実はセキュリティホール。
    • 本来はtext/plainで表示されるだけのJSコードが実行されたり。

Status Code

  • サーバからの応答を示す3桁の数字。
  • 100番台: 処理中の情報の伝達。
  • 200番台: 成功時の応答。
  • 300番台: サーバからクライアントへの命令。リダイレクトやキャッシュ利用の指示など。
    • リダイレクトさせるには、サーバからLocationヘッダーを返す。
    • 301: ドメイン移転、HTTPS化。
    • 302: 一時メンテナンス。
    • 303: ログイン後のページ遷移。
  • 400番台: リクエストに不正がある場合の応答。
  • 500番台: サーバ内部でエラーが発生したときの応答。

URL

  • Uniform Resource Locators
  • 構造:
    • {スキーム}://{ホスト名}/{パス}
    • {スキーム}://{ユーザ}:{パスワード}@{ホスト名}:{ポート}/{パス}?{クエリー}#{フラグメント}
  • ホスト名は大文字小文字の区別なし。
  • HTTPの仕様上URLの長さに制限はないが、ブラウザによってはあるのでだいたい2000文字までにしておくとよい。

ボディ

  • ヘッダーとの間に空行を挟んだ以降がすべてボディとなる。
  • GETメソッド時にリクエストボディを含めても、サーバ側がボディを解釈しなければ意味はない。

2章 HTTP/1.0のセマンティクス : ブラウザの基本機能の裏側

URLエンコーディング

  • RFC1866
    • ブラウザが使用。
    • [a-zA-Z0-9], *, -, ,, _以外をエスケープする。
    • は+になる。
  • RFC3986
    • パーセントエンコーディングとも。
    • curl --data-urlencodeが使用。
    • は%20になる。

マルチパートフォーム形式

  • 1度のリクエストで複数のファイルを送信。
  • リクエストボディが以下のようなブロックで構成。
    • boundary
    • header
    • content

コンテントネゴシエーション

  • 1つのリクエストの中でクライアントとサーバが互いに妥協可能な内でベストな設定を共有する。
リクエストヘッダー レスポンス 対象
Accept Content-Type MIMEタイプ
Accept-Language Content-Language / HTMLタグ 表示言語
Accept-Charset Content-Type 文字のキャラクターセット
Accept-Encoding Content-Encoding ボディの圧縮形式

クッキー

  • ウェブサイトの情報をブラウザ側に保存する。
  • サーバがクライアントにクッキーの保存を指示する。
  • ヘッダーを使う。
    • サーバ : Set-Cookie: {name}:{value}
    • ブラウザ : Cookie: {name}:{value}
  • 目安の最大容量は4KB。
分類
  • 発行元
    • ファーストパーティ : ユーザが訪れたサイト
    • サードパーティ : それ以外のサイト
  • 使用期間
    • セッションクッキー : ブラウザを閉じたら消える
    • 永続クッキー : 期限が設定されていてブラウザを閉じても残る
  • 目的
    • 厳密に必要なクッキー : ECサイトの買い物かごなど
    • 環境設定クッキー : 言語、ユーザ名、パスワードなど
    • 統計クッキー : アクセスしたページやクリックしたリンク
    • マーケティングクッキー : ユーザの行動を追跡する
属性
  • Expires
    • クッキーの寿命。指定した年月日時分秒で無効化される。
  • Max-Age
    • クッキーの寿命。現在時刻から指定された秒数を足した時間に無効化される。
  • Domain
    • クライアントからクッキーを送信するサーバ。
  • Path
    • クライアントからクッキーを送信するサーバのパス。
  • Secure
    • HTTPS以外の接続時には、クライアントからクッキーを送信しない。
  • HttpOnly
    • JavaScriptからクッキーにアクセスできなくなる。
  • SameSite
    • 同じオリジンのドメインに対してクライアントからクッキーを送信。
    • Strict : 異なるオリジンには送信しない。
    • Lax : POSTメソッドによる遷移やリソース読み込み時には送信しない。
    • None : 特に制約なく、異なるオリジンにも送信する。

オリジン

  • 同一のオリジン（サイト）
    • スキーム、ドメイン（ホスト名）、ポートの3つが同じであること。
    • ファイルパスは異なってもよい。
    • ex. http://example.com と http://www.example.com は異なるオリジン。

認証

  • だれだテメー

認可

  • なにしてんだテメー
BASIC認証
  • base64({ユーザ名}:{パスワード})
  • SSL/TSL通信でないと簡単に漏洩する。
Digest認証
  • realm, nonce, algorithm, qopとかのアレ。
  • 計算量多い。
クッキーを使ったセッション管理
  • 現在最もポピュラー。
  • クライアントはフォームでIDとパスワードを送信。
    • 平文なのでSSL/TLS必須。
  • サーバはセッショントークンを発行。トークンをDBなどに保存。
  • トークンをクッキーとしてクライアントに返す。
  • 2度目以降のアクセスからはクッキーを再送させ、ログイン済みか確認。

プロキシ

  • HTTPなどの通信を中継。
  • 必要に応じてコンテンツを改変したり、サーバの代わりに応答する。
  • リクエストの開始行
    • 通常 : {メソッド名} {パス形式}
    • プロキシを設定 : {メソッド名} {URL形式}
  • X-Forwarded-For, Forwardedリクエストヘッダー
    • 送信元IPアドレスの履歴
    • X-Forwarded-For: {client}, {proxy1}, {proxy2}

ゲートウェイ

  • 通信内容はそのまま転送する。
  • クライアントからはその存在に気づかれてはならない。

キャッシュ

  • クライアント側にすでにダウンロード済みで内容に変化がないコンテンツの、余計なダウンロードを抑制。
  • ブラウザ、プロキシ、CDNなどで保存される。
更新日時によるキャッシュ
  • サーバ -> クライアント
    • Last-Modifiedヘッダー
  • クライアント -> サーバ
    • If-Modified-Sinceヘッダー
  • サーバ -> クライアント
    • 更新あり : 200 OK。更新後のコンテンツをボディに含める。
    • 更新なし : 304 Not Modified。ボディなし。
Expiresヘッダー
  • レスポンスのヘッダー。日時が格納される。
  • 現在日時がこの期限内であれば、クライアントは強制的にキャッシュを利用し、リクエストを送信しなくなる。
Pragma: no-cache
  • リクエストヘッダー。
  • リクエストしたコンテンツがプロキシにキャッシュされていても、サーバにリクエストを送信させる。
  • HTTP1.1でCache-Controlヘッダーに統合。
確実にキャッシュしない条件
  • GET, HEAD以外のメソッド。
  • Cache-Control: private
  • Cache-Control: no-store
  • Authorizationヘッダーはあるが、Cache-Controlヘッダーにpublicがない。
Cache-Controlヘッダー
  • レスポンスヘッダー。Expiresヘッダーよりも優先される。
  • あくまでリロードを抑制するためで、プライバシー保護には使えない。
  • public
    • 同一コンピュータの複数ユーザ間でキャッシュを再利用するのを許可。
  • private
    • 同一コンピュータの別ユーザ間でキャッシュを再利用させない。
  • max-age={n}
    • キャッシュの鮮度を秒数で指定。
  • no-cache
    • キャッシュが有効かどうか毎回サーバに問い合わさせる。max-age=0とほぼ同じ。
  • no-store
    • キャッシュしない。
  • immutable
    • コンテンツが決して変化しないことを伝える。
Varyヘッダー
  • レスポンスヘッダー。
  • 同じURLでもクライアントによってレスポンスが異なることを示す。
  • ex. Vary: User-Agent, Accept_language
  • キャッシュを間違えて使わせないようにする。

Refererヘッダー

  • リクエストヘッダー。
  • リンク先のサイトにリンク元のURLを知らせる。
  • Webサービスの設計者は、プライバシー情報がGETパラメータとして表示されるような仕様を作成してはいけない。
    • Refererを通じて外部サービスに送信されるため。

検索エンジン（クローラー）向けのコンテンツのアクセス制御

robots.txt
  • クローラーに対してアクセスの許可・不許可を伝えるプロトコル。
  • HTMLのmetaタグや、HTTPのX-Robots-Tagでより細かい制御も可能。
  • これらを無視するクローラーを作成して実行しちゃったら、裁判にかけられるかも。
サイトマップ
  • サイトに含まれるページ一覧とそのメタデータを提供するxmlファイル。
  • JSで作られた動的ページからのリンクにも対応。

ユーザーエージェント

  • 通信を代行するプログラム。
    • ブラウザ、curl、HTTPクライアントライブラリなど。
