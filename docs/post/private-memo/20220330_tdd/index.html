まえがき

  • TDDのルール
    • 自動化されたテストが失敗したときのみ、新しいコードを書く。コードを書く前に自動テストコードを書く。
    • 重複を除去する。
  • ルールに従った際のプログラミングの順序
    1. レッド : 動作しない、コンパイルも通らないテストを1つ書く。
    2. グリーン : そのテストを迅速に動作させる。この時点ではクソコードでも可。
    3. リファクタリング : テストを通すために発生した重複をすべて除去する。

第1部 多国通貨

第1章 仮実装

  • 実装できそうな簡単なところからテストコードを書く。通常のプログラミングと同じ。
  • TDDサイクルのフェーズ
    1. 小さいテストを1つ書く。
    2. すべてのテストを実行し、1つ失敗することを確認する。
    3. 小さい変更を行う。実装に時間がかかりそうなコードはTODOリストにメモしておく。
    4. 再びテストを実行し、すべて成功することを確認する。
    5. リファクタリングを行い、重複を除去する。

第2章 明白な実装

  • コードに対する不快感などの感情をテストに翻訳する。
    • 副作用など。

第3章 三角測量

  • 三角測量
    • 2つ以上のテストコードから、プロダクトコードを一般化する手法。
    • 設計のアイデアやリファクタリング方法がわからない場合に有効。

第4章 意図を語るテスト

第5章 原則をあえて破るとき

  • TDDサイクルの1～4まではクソコード（ex. コピペ）でいいし設計の原則を無視してもよい。速度を重視する。
  • ただし、5で直す。

第6章 テスト不足に気づいたら

  • さっさと追加する。テストが不足していると、設計が腐敗しリファクタリングのコストが上昇し、プロジェクト進行が詰まっていく。

第7章 疑念をテストに翻訳する

  • 疑問に感じたらテストケースに反映させる。
  • さらなる設計は、本当に必要になるときまで先延ばしする。

第8章 実装を隠す

第9章 歩幅の調整

  • 小さい変更を行う途中で別の変更を割り込ませてもよい。ただし、割り込みは一度だけ。

第10章 テストに聞いてみる

  • コードの変更案を思いついたときは、悩むのではなく実際に変更してみてテストを実行してみればよい。

第11章 不要になったら消す

  • プロダクトコードに限らず、過剰なテストコードも消してよい。

第12章 設計とメタファー

第13章 実装を導くテスト

第14章 学習用テストと回帰テスト

  • 学習用テスト
    • 言語仕様を調べたり、チーム外の誰かが書いたソフトウェア（ライブラリなど）の機能を初めて使う際に書くテスト。
  • 回帰テスト
    • バグを再現させるためのテスト。
    • 本来はプロダクトコードを書いたときに同時に書かれるべきだったテスト。
    • これが通れば、そのバグは修正されたことになる。

第15章 テスト任せとコンパイル任せ

第16章 将来の読み手を考えたテスト

  • TDDでは、テストコードとプロダクトコードの量が同じくらいになる。
    • これまでの2倍コードを書けるようになるか、半分のコード量で同じだけの機能を書けないと、TDDのメリットを得られない。

第17章 多国通貨の全体ふりかえり

  • TODOリストが空のときは設計を見直す良いタイミングになる。
  • TDDは以下の代用にはならない。
    • パフォーマンステスト
    • 負荷テスト
    • ユーザビリティテスト

第2部 xUnit

第18章 xUnitへ向かう小さな一歩

第19章 前準備

  • テストを書くときのパターン
    • 準備 : オブジェクトを作る。
    • 実行 : そのオブジェクトに対して操作を行う。
    • アサート : 結果の検証を行う。
  • テスト間には絶対に依存関係を作ってはならない。
    • テストの順番が変わったら結果が変わるようなテストは駄目。
