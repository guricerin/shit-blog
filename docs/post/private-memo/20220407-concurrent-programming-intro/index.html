1章 並行性と並列性

  • モノ : ある特定の時間で全体が存在する。
  • プロセス : 特定の時間では部分しか存在しない。

1.1 プロセス

計算を実行する主体。

  1. 実行前状態
  2. 実行状態
  3. 待機状態
  4. 終了状態
  • 待機状態が必要な理由
    • データの到着を待つ。
    • 計算リソースの空きを待つ。
    • 自発的に待機状態になり、計算リソースの占有を防ぐ。

1.2 並行性

2つ以上のプロセスが同時に計算を進めている状態。
時刻tにおいて、ある複数のプロセス p1, …, pn が計算途中状態（実行状態 or 待機状態）にある。

  • OSプロセス : カーネルから見たプロセス。独立した仮想メモリ空間を割り当てられる。
  • スレッド : OSプロセス内に内包されるプロセス。所属するOSプロセスのリソースをスレッド間で共有する。

1.3 並列性

同じ時刻で複数のプロセスが同時に計算を実行している状態。
時刻tにおいて、ある複数のプロセス p1, …, pn が実行状態にある。

タスク並列性

複数のタスク（プロセス）を複数のCPU上で同時に実行すること。スレッド並列性とも。
マルチコアCPUやメニコアCPUの登場により、ソフトウェア側でも意識しなければならなくなった。

データ並列性

複数に分割したデータに対して並列に処理を行うこと。
ex. ベクトル演算

応答速度とスループット
  • 応答速度
    • 計算開始から終了までの期間。
    • 消費CPUクロック数 / CPU動作クロック周波数 などで算出した時間で表す。
  • スループット
    • 単位時間あたりに実行可能な計算量。
    • MIPS : 1秒間に何百万回のCPUインストラクションが実行可能かを表す単位。
    • FLOPS : 1秒間に何回の浮動小数点演算が実行可能かを表す単位。
アムダールの法則
  • 並列化可能な処理の実行時間とオーバヘッドの比に応じて実行速度が変わること
  • 並列化することでどの程度応答速度が向上するか
    • 並列化可能な部分の処理と不可能な処理の部分（前処理、後処理）との割合
    • 並列化に伴うオーバーヘッド（スレッド生成や同期処理）
  • 割合によっては、並列化するほうが遅くなることも。

インストラクション並列性

インストラクション（CPUの命令語）レベルで並列化を行う手法。
主にハードウェアやコンパイラが暗黙的に行う。プログラマが行うことはほぼない。

  • インストラクションを以下のように分割し、複数のインストラクションがバケツリレーのように並列に実行可能とする。
    • 命令読み込み
    • 命令解釈
    • 実行
    • メモリアクセス
    • レジスタ書き込み

1.4 並行と並列処理の必要性

並列処理と性能向上

  • 半導体素子の微細化により
    • シリコンウェハー上に作成可能な回路数が増え、面積あたりに多くの回路が作成可能。
    • ウェハーあたりに作成可能なダイ（CPUのチップなど）が増える。
    • ダイの製造単価が下がる。
    • 半導体で作成されたトランジスタのオンオフが高速化。
  • ただし、
    • CPUの動作周波数を上げると消費電力と発熱が多くなる。
    • 半導体素子の劣化が激しくなり、動作が不安定となる。
    • 電圧を下げることで、消費電力と発熱に対処。
    • が、微細化が進み過ぎたことで半導体トランジスタにおけるリーク電流が問題となる。
    • これは量子トンネル効果が原因なので、対処するには半導体素子を大きくするか電圧を上げる必要あり。本末転倒。
    • 諸問題がどれも限界に達した結果、マルチコアやメニコアCPUへと進化（2000年代中頃）。ソフトウェア側で並列性を意識せざるを得ない。

並行処理の必要性と計算パス数爆発

プロセスがn個あった場合、実際にとりうる実行順（計算パス）はn!となる。
もしn!のうち数パターンだけにバグが存在する場合、きわめて再現性の低い厄介なバグとなる。
そのため、並列プログラミングの技術のみで並行プログラミングを行うことは良くない。

2章 プログラミングの基本

2.1 アセンブリ言語

  • AArch64 : Armの64ビットCPUアーキテクチャ。
  • x86-64 : AMDとIntel CPUのCPUアーキテクチャ。

2.2 C言語

Pthreads

POSIX標準のインターフェイスを備えるスレッドライブラリの総称。

volatile修飾子

  • コンパイラは遅いメモリアクセスを抑制するためにレジスタにいったんコピーしてから値を利用する。
  • この最適化は、メモリ上の値を監視したりするときに障害になるので、最適化を抑制するvolatile修飾子が有用となる。
  • 複数のプロセスから同一メモリにアクセスしたい場合に、コンパイラの最適化によりメモリアクセスが行われなくなると問題が起きる。

2.3 Rust

所有権

  • 古典的論理
    • A, A -> B, A -> C |> B & C
    • 「Aである」, 「AならばBである」、「AならばC」が成り立つとき、BかつCであると推論可能。
  • 線形論理
    • リソースの利用（ここではA）に制限を持たせることのできる論理体系。
    • A, A -o B, A -o C |> B
    • or
    • A, A -o B, A -o C |> C
    • 「Aである」, 「AならばBである」、「AならばC」が成り立つとき、BまたはCであると推論可能。
    • 線形型はこの線形論理を元にしており、線形型の姉妹分であるアフィン型を適用したRustでは所有権の移動に活用されている。

ライフタイム

  • Rustのライフタイムは型の一種。
  • 異なるライフタイムでも、範囲が狭い（寿命が短い）ほうに合わせる。これは部分型付けで実現している。

3章 同期処理1

  • 同期処理 : 並行処理において、タイミングの同期やデータ更新などを複数プロセス間で協調的に行うこと。
  • Rustの同期処理ライブラリは裏側でPthreadsを使用している。

3.1 レースコンディション

  • レースコンディション : 複数プロセスが並行して共有リソースにアクセスした結果引き起こされる、予期しない異常な状態。
  • クリティカルセクション : レースコンディションを引き起こすコード部分。

ex.
共有変数vを読み込み、インクリメントする2つのプロセス。
インクリメントするタイミングが重なると、正しくインクリメントされない（後からインクリメントした処理がなかったことになる）。

3.2 アトミック処理

不可分操作。それ以上は分割不可能な処理。

  • その処理の途中状態はシステム的に観測できず、かつ、もしその処理が失敗した場合は処理前の状態に復元される。
  • 最近のCPUではアトミック処理用の命令がサポートされており、現代の同期処理のほとんどはこれらに依存している。

Compare and Swap (CAS)

アトミック処理の一種。セマフォ、ロックフリー、ウェイトフリーなデータ構造を実装するために利用される。
書き込みしたいデータのアドレス A、元のデータ B、新たなデータ Cを渡し、*A == B の場合のみ *A = C とする処理。

Test and Set (TAS)

アトミック処理の一種。スピンロックなどを実装するために利用される。
bool型のポインタ p を渡し、*p = trueならなにもせずtrueを返す。*p = false なら、*p = trueにしてfalseを返す。

Load-Link / Store-Conditional (LL/SC)

  • lock命令プリフィックス : x86, x86-64のCPUアーキテクチャにおけるメモリアクセスの排他処理のための命令。
  • LL/SC : ARM, RISC-V, POWER, MIPSなどのCPUにおけるアトミック処理のための命令。
    • LL : 読み込み。
    • SC : 他のCPUによって書き込みされていない場合のみ書き込み成功。
    • lock命令プリフィックスとは違って、他のCPUからの書き込みがあったかどうかを検知可能。

3.3 ミューテックス

MUTtual EXclusion (mutex)。排他実行。
クリティカルセクションを実行可能なプロセス数を1つに制限するような同期処理。
TASを用いることでlock変数からの読み込みと書き込みが同時に行えるようにしている。

スピンロック

リソースの空きをポーリングして確認するようなロックの獲得方法。
典型的には、スピンロック用のAPIはロック獲得用とロック解放用の関数の二つが提供される。

  • TTAS : 検査をしてからTASを呼び出すことで、重いアトミック命令の回数を減らす。

Pthreadsでのミューテックス

リソースを消費しがちなので、スピンロック単体での使用は推奨されない。Pthreadsや言語ライブラリで提供されるミューテックスか、スピンロックとこれらでの組み合わせを利用すべき。

3.4 セマフォ

semaphore。
ロック獲得可能なプロセス数は最大1個のミューテックスに対して、セマフォでは最大n個のプロセスが同時にロック獲得可能。
つまり、ミューテックスはセマフォの特殊なケース。
ミューテックスとは違って、レースコンディションは防げない。

POSIXセマフォ

セマフォの標準的実装。

  • 名前付きセマフォ
    • ファイルで共有リソースを指定可能。
    • メモリを共有しないリソースでも用意にセマフォを実現可能。
  • 名前なしセマフォ
    • 共有メモリ領域が必要となる。

3.5 条件変数

ある条件が満たされない間はプロセスを待機しておき、条件が満たされた場合に待機中のプロセスを実行する手法。
Pthreadではpthread_cond_t型と関数を利用する。

  • 重要な点
    • 条件変数へのアクセスは必ずロック獲得後に行わなければならない。
    • pthread_cond_t型の条件変数以外にも、実行可能かを表す条件変数を用意しなければならない。

3.6 バリア同期

すべてのスレッドが揃ってから、処理の続きを実行といった同期手法。

スピンロックベースのバリア同期

待機中にもループ処理を行うため、無駄にCPUリソースを消費してしまう。

Pthreadsを用いたバリア同期

Pthreadsの条件変数を用いる。

3.7 Readers-Writerロック

読み込みのみを行うプロセス（Reader）と読み書きのみを行うプロセス（Writer）に分類し、以下の制約を満たすように排他制御を行う。

  • ロック獲得中のReaderが同時刻に複数（0以上）存在可能。
  • ロック獲得中のWriterは同時刻に最大1つのみ存在可能。
  • ReaderとWriterが同時刻にロック獲得状態にならない。

スピンロックベースのRWロック

RWロックは書き込みはほとんど行わない場合に向いている。書き込みが頻繁に起きるならミューテックスのほうが実行速度と安全性の面からよい。

PthreadsのRWロック

3.8 Rustの同期処理ライブラリ

Rustの同期処理ライブラリは、クリティカルセクション外での保護対象オブジェクトのアクセスと、ロック解除忘れを型システムによって防ぐようになっている。

ミューテックス

std::sync::Mutex型。std::sync::Arcなどのスレッドセーフなスマートポインタに包む。

条件変数

std::sync::Condvar型。std::sync::Arcなどのスレッドセーフなスマートポインタに包む。

RWロック

std::sync::RwLock型。

バリア同期

std::sync::Barrier型。std::sync::Arcなどのスレッドセーフなスマートポインタに包む。

セマフォ

Rustにおいて標準では用意されていないが、ミューテックスと条件変数を用いて実装可能。
セマフォを用いると、キューのサイズが有限なチャネルを実装可能。

  • チャネル
    • プロセス間でメッセージ交換を行うための抽象的な通信路。
    • Rustでは、チャネルは送信端と受信端に分かれている。
    • Sendトレイとを実装している型のみがチャネルを介して送受信可能。
    • Sendトレイトを実装していない型 : Rc型（非スレッドセーフな参照カウントのスマートポインタ）

3.9 パン屋のアルゴリズム

CPUが提供するアトミック命令を利用しない同期処理手法。
他にはデッカーのアルゴリズム、ピーターソンのアルゴリズムなど。
パン屋というよりは病院や役所、レストランなどでの整理券のメタファー。

4章 並行プログラミング特有のバグと問題点

4.1 デッドロック

  • デッドロック

    • お互いにリソースの空きを待ち、処理が進まない状態。

  • 食事する哲学者問題

    • 円卓に哲学者が座っており、哲学者の前には食事が、哲学者の間には箸が1本ずつ置かれているとする。
    • 哲学者は箸を2本取り上げた際に食事を行った後、箸を元の位置に戻す。
    • タイミングによっては箸を1本ずつ持った哲学者が互いにもう一本の箸を待つ状態になる。

  • デッドロックとなる状態機械

    • 状態機械がデッドロックする可能性がある <-> 初期状態から到達可能かつ次の遷移先がないような状態を持つ

4.2 ライブロックと飢餓

  • 食事する哲学者問題の改良版

    • まず左の箸の空きを待ち、空いたら取り上げる。
    • 右の箸の空きを待つ。しばらくしても空かなかったら左の箸を元に戻し、はじめに戻る。
    • こうしても、タイミングによっては2人の哲学者が同時に左の箸を取り（相手視点では右側の箸）、デッドロックが発生する。

  • ライブロック

    • リソース獲得の処理は行われるが、リソースが獲得できずに次の処理が行われない状態。

  • ライブロックとなる状態機械

    • 状態機械がライブロックする可能性がある <-> 常にいずれかのリソース獲得状態へ到達可能だが、それら状態へは決して到達しないような無限の遷移列が存在する

  • 飢餓

    • 特定のプロセスのみがリソース獲得状態へ遷移しないような状態。

  • 飢餓を引き起こす状態機械

    • 状態機械が飢餓を引き起こす可能性がある <-> あるプロセスが存在して、常にそのプロセスのリソース獲得状態へ到達可能だが、その状態へは決して到達しないような無限の遷移列が存在するか、デッドロックとなる状態機械である

4.3 銀行家のアルゴリズム

  • 銀行家のアルゴリズム
    • by ダイクストラ
    • リソース割り当ておよびデッドロック回避のアルゴリズム。
    • デッドロックとなる状態に遷移するかをシミュレーションにより判定してからリソース割り当てを行う（あるいは行わない）。
    • 欠点 : 動作するスレッド数と、各スレッドが必要なリソースの最大値を事前に把握しておく必要あり。

4.4 再帰ロック

  • 再帰ロック
    • ロックを獲得中のプロセスが、そのロックを解放前に、再度そのロックを獲得すること。
    • 再入可能 : 再帰ロックを行っても処理が続行可能であること。
      • 単純なミューテックスに対して再帰ロックを行うとデッドロックが起きる。
    • Rustの再帰ロックは未定義動作。
    • RustのArcは、他のスレッドに共有リソースを渡す場合にのみcloneし、同一スレッド内ではcloneして利用しない、ということを守ればよい。

4.5 疑似覚醒

  • 疑似覚醒
    • ある状態が満たされるまで待機中のプロセスが、条件が満たされていないにもかかわらず実行状態へ移行すること。
    • 典型例 : wait中のシグナルによる割り込みが原因で発生する。
    • 環境依存の動作のため、waitする際は自前の条件チェックを行うほうがよい。

4.6 シグナル

  • シグナルとマルチスレッドは相性が悪い。
    • どのタイミングでシグナルハンドラが呼び出せるかわからないため。
    • シグナルハンドラの中に（メインスレッドと同じミューテックスの）ロックを獲得する処理を入れると、デッドロックしがち。
  • シグナル受信専用のスレッドを用いることで、上記の状況を回避する。

4.7 メモリバリア

  • アウトオブオーダ実行
    • 現代的なCPUでは必ずしも機械語の命令順に処理は行うわけではない。
    • 理由 : パイプライン処理時にIPS（instruction per second, 単位時間あたりの実行命令数）を向上させるため。
  • メモリバリア（メモリフェンス）
  • アウトオブオーダによって引き起こされる問題から保護するための処理。
  • 問題 : 共有変数に対するread命令がロック獲得処理より前に実行されることによるレースコンディション。
  • 機械語には、メモリバリアが組み込まれたメモリ読み書き命令がある。
